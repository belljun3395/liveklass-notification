== 개요

`NotificationJob`은 알림 발송의 단위입니다.
Job은 7개의 상태를 가지며, UseCase/Scheduler/Processor가 `JobNotificationPolicy`를 통해 상태를 전이합니다.

이 문서는 **누가(실행 주체)** 어떤 **정책(Policy)**을 사용하여 상태를 전이하는지를 종합적으로 기술합니다.

== Job 상태 설명

[cols="2,1,4", options="header"]
|===
| 상태 | 유형 | 설명
| `CREATED` | 초기 | Job이 생성된 초기 상태. 발송 시각 확정 전
| `SCHEDULED` | 진행 | 발송 시각이 확정되어 스케줄러에 등록된 상태
| `PROCESSING` | 진행 | 알림 발송이 실제로 진행 중인 상태
| `RETRYING` | 진행 | 실패/취소 후 재처리를 준비 중인 상태
| `CANCELLED` | 취소 | Job이 취소된 상태. RETRYING으로 재시도 가능
| `FAILED` | 오류 | 처리 중 실패한 상태. RETRYING으로 전이 가능
| `COMPLETED` | 종료 | 모든 알림 발송이 완료된 최종 상태 (terminal)
|===

== JobNotificationPolicy (정책)

`JobNotificationPolicy`는 **Job 상태 전이**와 그에 수반되는 **Notification 상태 전이**를 하나의 enum 상수로 선언합니다.
호출부가 정책 상수를 명시적으로 선택하며, 런타임에서 동적 조회는 하지 않습니다.

=== 정책의 이중 역할

정책은 두 가지 역할을 동시에 수행합니다:

1. **Job 상태 전이 게이트**: `transitionTo(policy)` — `fromStatuses`에 현재 상태가 포함되면 `toStatus`로 전이
2. **Notification 전이 허용 목록**: `assertNotificationTransitionAllowed(policy, from, to)` — 해당 정책이 적용되는 **Job 상태 구간 전체**에서 허용되는 Notification 전이를 선언

특히 `TO_PROCESSING`은 Job이 PROCESSING 상태인 **동안** Orchestrator가 수행하는 전체 발송 파이프라인(PENDING → SENDING → SENT/FAILED → RETRY_WAITING/DEAD_LETTER)의 Notification 전이를 모두 포함합니다.

=== 정책 목록

[cols="2,2,2,4", options="header"]
|===
| 정책 | Job From | Job To | Notification 전이

| `TO_SCHEDULED`
| CREATED
| SCHEDULED
| 없음

| `TO_PROCESSING`
| SCHEDULED, RETRYING
| PROCESSING
| _Job 전이 시점:_ +
PENDING -> SENDING +
RETRY_WAITING -> SENDING +
 +
_PROCESSING 구간 내 발송 파이프라인:_ +
SENDING -> SENT +
SENDING -> FAILED +
FAILED -> RETRY_WAITING +
FAILED -> DEAD_LETTER

| `TO_CANCELLED`
| CREATED, SCHEDULED, RETRYING
| CANCELLED
| PENDING -> CANCELLED +
RETRY_WAITING -> CANCELLED +
FAILED -> CANCELLED +
DEAD_LETTER -> CANCELLED +
 +
NOTE: SENDING 상태 Notification은 이미 외부 채널에 발송 요청이 나간 상태이므로 취소 대상에서 제외됩니다. +
RETRYING 상태에서 취소 시, SENDING Notification은 존재하지 않습니다 (RETRYING 진입 전 SENDING → RETRY_WAITING 롤백이 완료되므로).

| `TO_RETRYING`
| FAILED, CANCELLED
| RETRYING
| DEAD_LETTER -> RETRY_WAITING +
CANCELLED -> RETRY_WAITING

| `TO_COMPLETED`
| PROCESSING
| COMPLETED
| 없음

| `TO_FAILED`
| PROCESSING
| FAILED
| 없음

| `STUCK_RECOVERY`
| PROCESSING
| FAILED
| SENDING -> DEAD_LETTER

| `STUCK_SENDING_ROLLBACK`
| PROCESSING
| PROCESSING
| SENDING -> RETRY_WAITING
|===

=== 정책이 적용되는 방식

`NotificationJob`은 두 가지 방식으로 정책을 사용합니다.

1. **Job 상태 전이** -- `transitionTo(policy)`: `policy.fromStatuses`에 현재 상태가 포함되면 `policy.toStatus`로 전이. 포함되지 않으면 `false`를 반환하고 전이하지 않음
2. **Notification 상태 전이 검증** -- `assertNotificationTransitionAllowed(policy, from, to)`: 정책에 선언된 `NotificationTransition` 집합에 해당 전이가 포함되는지 확인. 포함되지 않으면 `IllegalStateException` 발생

[source,java]
----
// Job 상태 전이 — 실패 시 false 반환 (예외 없음)
public boolean markProcessing() {
    return transitionTo(JobNotificationPolicy.TO_PROCESSING);
}

// Notification 상태 전이 검증 — 실패 시 IllegalStateException
public void startSendingNotification(Notification n) {
    assertNotificationTransitionAllowed(
        JobNotificationPolicy.TO_PROCESSING,
        n.getStatus(), NotificationStatus.SENDING);
    n.markSending();
}
----

=== UseCase (사용자 요청)

[cols="2,2,2,3", options="header"]
|===
| 클래스 | 전이 | 정책 | 비고

| `CreateScheduledNotificationJobUseCase`
| -> CREATED
| (팩토리)
| Job + Notification(PENDING) 생성, `NotificationJobCreatedAfterCommitEvent` 발행. +
특수 락 해제 순서: TX 커밋 전에 락 해제 — AfterCommit 핸들러가 같은 키로 락을 잡을 수 있도록 함

| `CancelNotificationJobUseCase`
| CREATED/SCHEDULED/RETRYING -> CANCELLED
| `TO_CANCELLED`
| 분산 락 + SELECT FOR UPDATE. +
Notification 일괄 취소(PENDING/RETRY_WAITING/FAILED/DEAD_LETTER -> CANCELLED). +
`NotificationJobScheduleEvent(CANCEL)` 발행으로 스케줄 해제

| `RecoverNotificationJobUseCase`
| FAILED/CANCELLED -> RETRYING
| `TO_RETRYING`
| 분산 락 + SELECT FOR UPDATE. +
FAILED → Notification 리셋(DEAD_LETTER -> RETRY_WAITING). +
CANCELLED → Notification 리셋(CANCELLED -> RETRY_WAITING). +
같은 트랜잭션에서 처리하여 RetryingScheduler 선점 레이스 방지

[cols="2,2,2,3", options="header"]
|===
| 클래스 | 전이 | 정책 | 비고

| `DbScheduledNotificationJobRelay`
| SCHEDULED -> PROCESSING
| `TO_PROCESSING`

=== Schedule (주기적 폴링 / 기동 시 복구)

[cols="2,2,2,3", options="header"]
|===
| 클래스 | 전이 | 정책 | 비고

| `RetryingNotificationJobExecutionScheduler`
| RETRYING -> PROCESSING
| `TO_PROCESSING`
| RETRYING Job 폴링. +
전이 후 `NotificationJobStatusChangedEvent` + `NotificationJobExecutionEvent` 발행. +
분산 락 + SELECT FOR UPDATE

| `ProcessingNotificationJobExecutionScheduler`
| --
| --
| PROCESSING Job 폴링. +
Job 상태 전이 없음, `NotificationJobExecutionEvent` 발행. +
분산 락 미사용 — `NotificationJobExecutionProcessor`가 락으로 중복 실행 방지

| `StuckProcessingRecoveryScheduler`
| PROCESSING -> FAILED 또는 PROCESSING 유지
| `STUCK_RECOVERY` +
`STUCK_SENDING_ROLLBACK`
| 타임아웃 초과 PROCESSING Job 감지. +
잔류 SENDING 알림별: +
  `sendTryCount < max` → RETRY_WAITING(롤백, `STUCK_SENDING_ROLLBACK`) +
  `sendTryCount >= max` → DEAD_LETTER(최종 실패, `STUCK_RECOVERY`) +
롤백된 알림이 있으면 Job은 PROCESSING 유지 (다음 실행 사이클에서 재처리). +
처리 가능한 알림(PENDING/RETRY_WAITING)이 없을 때만 Job → FAILED 전이

| `NotificationJobRestartReconciler`
| --
| `STUCK_SENDING_ROLLBACK`
| `ApplicationReadyEvent` 수신, 기동 시 1회 실행. +
PROCESSING Job의 잔류 SENDING -> RETRY_WAITING 롤백. +
Job 상태는 PROCESSING 유지. +
분산 락 미사용, SELECT FOR UPDATE만 사용 — 롤링 배포 시 여러 인스턴스가 동시에 기동하면 SELECT FOR UPDATE로 직렬화되어 기능적 오류 없음
|===

== 동시성 제어

=== 방어 메커니즘

모든 상태 전이 주체는 다음 패턴을 조합하여 중복 처리를 방지합니다:

1. **분산 락** (`DistributedLock.tryLock(idempotencyKey, ttl)`) -- 다중 인스턴스 간 상호 배제
2. **SELECT FOR UPDATE** (`findByIdAndDeletedFalseForUpdate`) -- DB 수준 행 잠금으로 동시 전이 차단
3. **상태 재검증** -- 잠금 획득 후 현재 상태가 전이 가능한지 재확인 (예: `markProcessing()` 실패 시 스킵)
4. **SENDING 선점 저장** -- 발송 전 Notification을 SENDING으로 먼저 커밋(`REQUIRES_NEW` TX)하여 다른 실행자가 같은 알림을 선택하지 못하게 함

[cols="3,1,1,1", options="header"]
|===
| 실행 주체 | 분산 락 | SELECT FOR UPDATE | Watchdog

| CreateScheduledNotificationJobUseCase | O | X | X
| CancelNotificationJobUseCase | O | O | X
| RecoverNotificationJobUseCase | O | O | X
| NotificationJobCreatedProcessor | O | X | X
| NotificationJobExecutionProcessor | O | O | O (TTL/3)
| DbScheduledNotificationJobRelay | O | O | X
| DbRetryScheduledNotificationRecoverRelay | O | O | X
| RetryingNotificationJobExecutionScheduler | O | O | X
| StuckProcessingRecoveryScheduler | O | O | X
| ProcessingNotificationJobExecutionScheduler | X | X | X
| NotificationJobRestartReconciler | X | O | X
|===

=== 중복 발송 방지 분석

이 시스템의 핵심 요구사항은 **같은 Notification이 두 번 발송되면 안 된다**는 것입니다.
아래는 중복 발송이 발생할 수 있는 시나리오와 현재 방어 구조를 분석한 결과입니다.

==== 현재 모델의 성격

현재 구조는 **exactly-once가 아닌 복구 가능한 at-least-once 발송 모델**입니다.

* **중복 실행 억제**는 잘 하고 있음
* 하지만 **중복 발송 절대 금지**를 외부 채널 수준에서 보장하지는 못함

가장 큰 이유는 **외부 발송 성공과 DB 상태 반영(SENT)이 원자적이지 않기 때문**입니다.

==== 시나리오 1: 외부 발송 성공 후 크래시 (위험도: 높음)

**상황:** Notification이 외부 채널로 실제 발송된 직후, DB에 SENT로 반영하기 전에 프로세스가 종료됩니다.

**흐름:**

[plantuml, concurrency-scenario-1, svg]
----
@startuml
participant "ExecutionProcessor" as EP
participant "SendOrchestrator" as SO
database "DB" as DB
participant "외부 채널\n(FCM 등)" as EXT
participant "RestartReconciler\n/ StuckRecovery" as REC

== 정상 발송 시작 ==
EP -> SO : execute(job)
SO -> DB : Notification → SENDING (REQUIRES_NEW 커밋)
SO -> EXT : 실제 발송
EXT --> SO : 성공 응답

== 크래시 발생 ==
SO ->x SO : 프로세스 종료\n(SENT 저장 전)

...서버 재시작...

== 복구 ==
REC -> DB : SENDING 잔류 감지
REC -> DB : SENDING → RETRY_WAITING (타임아웃으로 판단)
note right: 상태상 "미발송"이므로 재시도 가능으로 표시\n하지만 실제로는 이미 외부에 발송됨

...다음 실행 사이클...

EP -> SO : execute(job)
SO -> EXT : **동일 알림 재발송**
note right #FFCCCC: 외부 채널에서 중복 발송 발생\n(idempotency 보장 없으면)
@enduml
----

**방어 (시스템 레벨):** 없음. Job/Notification 상태 머신에서는 방지 불가능.

**방어 (채널 레벨):** 외부 발송 채널이 idempotency key를 지원하면 차단 가능. 예: 이메일 서비스의 `Message-ID`, FCM의 `send_id`.

**판단:** 구현 실수가 아니라 구조적 한계입니다. 외부 시스템과의 분산 트랜잭션 없이는 시스템 레벨 완전 방지 불가능. at-least-once 모델의 근본적 한계이므로, 외부 채널의 idempotency 보장에 의존합니다.

**추가 고려:** `NotificationSendingStatePersister`가 `REQUIRES_NEW` TX로 SENDING 상태를 먼저 커밋하므로, 크래시 시 SENDING 상태가 DB에 남아 복구 시스템이 감지할 수 있습니다. 단, 복구 시스템은 "발송 완료 후 크래시"와 "발송 전 크래시"를 구분할 수 없으므로, 두 경우 모두 RETRY_WAITING으로 롤백합니다.

==== 시나리오 2: 같은 Job에 대한 이중 실행 (위험도: 방어됨)

**상황:** `ProcessingNotificationJobExecutionScheduler`가 PROCESSING 상태의 Job에 대해 execution event를 반복 발행합니다. 두 ExecutionProcessor가 동시에 같은 Job을 처리하려 합니다.

**흐름:**

[plantuml, concurrency-scenario-2, svg]
----
@startuml
participant "Executor A" as A
participant "Executor B" as B
database "DB" as DB

== Executor A 선점 ==
A -> A : 분산 락 획득 (idempotencyKey)
A -> DB : SELECT FOR UPDATE (행 잠금)
A -> DB : Notification → SENDING\n(REQUIRES_NEW 커밋)
A -> A : 발송 시작...

== Executor B 진입 시도 ==
B -> B : 분산 락 시도
alt 락 보유 중
    B -> B : tryLock 실패 → 스킵
else 락 TTL 만료 (watchdog 갱신 실패)
    B -> B : 분산 락 획득
    B -> DB : SELECT FOR UPDATE → **대기**\n(A의 트랜잭션이 끝날 때까지)
    note right: @ApplicationModuleListener의\nREQUIRES_NEW TX 안에서\nFOR UPDATE + 발송이 실행되므로\n**Job row의 행 잠금**이\n발송 완료까지 유지됨
    A -> DB : SENT 저장 + TX 커밋\n(행 잠금 해제)
    B -> DB : 잠금 획득 → 상태 확인
    B -> DB : 타겟 조회: PENDING/RETRY_WAITING → **빈 목록**\n(모두 이미 SENT)
    B -> B : 발송 대상 없음 → 종료
end
@enduml
----

**방어:** 3단계로 방어됩니다.

1. **분산 락** -- 대부분의 중복 진입을 차단
2. **SELECT FOR UPDATE (Job row)** -- `NotificationJobExecutionProcessor`가 `findByIdAndDeletedFalseForUpdate`로 Job row에 행 잠금을 획득합니다. `@ApplicationModuleListener`의 REQUIRES_NEW 트랜잭션이 Orchestrator 실행(발송 포함)을 감싸므로, 발송 완료 + 결과 저장 + TX 커밋까지 행 잠금이 유지됩니다. 다른 실행자는 이 TX가 끝날 때까지 대기합니다.
3. **SENDING 선점 저장** -- `NotificationSendingStatePersister`가 REQUIRES_NEW TX로 Notification을 SENDING으로 즉시 커밋합니다. 따라서 행 잠금 해제 후 다른 실행자가 진입해도, 이미 SENDING/SENT인 알림은 대상에서 제외됩니다.

**행 잠금 대상 명확화:**

* `NotificationJobExecutionProcessor`의 `findByIdAndDeletedFalseForUpdate` → **Job row**에 행 잠금
* `NotificationSendingStatePersister`의 REQUIRES_NEW TX → **Notification rows**를 별도 TX에서 저장 (Job row 잠금과 독립)

**판단:** 현재 구조에서 이 시나리오는 안전합니다.

==== 시나리오 3: StuckRecovery와 활성 실행의 충돌 (위험도: 방어됨)

**상황:** Job이 `stuckTimeoutSeconds`를 초과했지만 실제로는 아직 발송 중입니다. `StuckProcessingRecoveryScheduler`가 이를 "스턱"으로 오판합니다.

**방어 구조:**

1. **분산 락 동일 키** -- ExecutionProcessor와 StuckRecovery 모두 같은 `idempotencyKey`로 락을 시도. 실행 중이면 Watchdog이 락을 갱신하므로 StuckRecovery가 `tryLock` 실패로 스킵
2. **락 TTL 만료 시** -- Watchdog 갱신이 실패하여 StuckRecovery가 락을 획득해도, SELECT FOR UPDATE에서 ExecutionProcessor의 트랜잭션이 끝날 때까지 대기. 트랜잭션 완료 후 상태를 확인하면 이미 COMPLETED/FAILED
3. **타임아웃 재검증** -- StuckRecovery는 SELECT FOR UPDATE 후 `updatedAt`을 다시 확인합니다. ExecutionProcessor의 TX가 완료되면서 `updatedAt`이 갱신되므로, 타임아웃 미초과로 판단하여 스킵

**판단:** 시나리오 2와 동일한 메커니즘으로 방어됩니다.

==== 시나리오 4: 스케줄 중복 처리 (위험도: 방어됨)

**상황:** `DbScheduledNotificationJobRelay`에서 두 인스턴스가 같은 스케줄 row를 읽고 각각 처리를 시도합니다.

**방어:** 2단계로 방어됩니다.

1. **분산 락** -- `DbScheduledNotificationJobRelay`는 `idempotencyKey`로 분산 락을 획득합니다. 첫 번째 인스턴스가 락을 점유하면 두 번째 인스턴스는 스킵
2. **SELECT FOR UPDATE + 상태 재검증** -- 분산 락 만료 시에도, `findByIdAndDeletedFalseForUpdate`로 행 잠금을 잡으므로 첫 번째 인스턴스가 SCHEDULED → PROCESSING 전이 후, 두 번째 인스턴스는 `markProcessing()` 실패(false 반환)로 스킵

Job 중복 전이와 중복 발송 모두 발생하지 않습니다.

==== 시나리오 5: Recover UC와 RetryingScheduler 레이스 (위험도: 없음)

**상황:** `RecoverNotificationJobUseCase`가 RETRYING으로 전이하는 동안 `RetryingNotificationJobExecutionScheduler`가 해당 Job을 감지합니다.

**방어:** RecoverUC는 `TransactionTemplate` 안에서 RETRYING 전이 + Notification 리셋을 원자적으로 처리하고, 분산 락을 **트랜잭션 커밋 이후에** 해제합니다. RetryingScheduler는 같은 키로 락을 시도하므로 UC 완료 전까지 진입 불가. 완료 후에는 Job(RETRYING) + Notification(RETRY_WAITING)이 모두 커밋된 상태입니다.

==== 시나리오 6: RestartReconciler 롤링 배포 중 동시 실행 (위험도: 낮음)

**상황:** 롤링 배포에서 여러 인스턴스가 거의 동시에 기동하여 각각 `ApplicationReadyEvent`를 수신합니다.

**방어:** `NotificationJobRestartReconciler`는 분산 락을 사용하지 않지만, `findByIdAndDeletedFalseForUpdate`로 **Job row에 행 잠금**을 획득합니다. 첫 번째 인스턴스가 SENDING → RETRY_WAITING 롤백을 완료하고 TX를 커밋하면, 두 번째 인스턴스는 잠금 획득 후 SENDING 알림이 빈 목록이므로 아무 작업 없이 종료합니다.

**판단:** SELECT FOR UPDATE로 직렬화되어 기능적 오류는 없지만, 분산 락 없이 DB 잠금에만 의존합니다. 기동 시 1회 실행이므로 실무적 위험은 낮습니다.

=== 트랜잭션 경계 설계

외부 서비스 호출(`NotificationJobSchedulePort`, `NotificationService`)은 원칙적으로 트랜잭션 밖에서 수행됩니다.
이 설계의 이유와 각 호출 지점의 실패 시 영향을 정리합니다.

==== 원칙

외부 서비스 호출이 트랜잭션 안에 있으면 두 가지 문제가 발생합니다:

1. **행 잠금 보유 시간 증가** -- 외부 호출 응답 대기 동안 FOR UPDATE 잠금이 유지되어 다른 실행 주체를 차단
2. **롤백 시 불일치** -- 외부 호출 성공 후 TX 롤백 시, 외부에는 반영됐지만 DB에는 반영되지 않은 상태 발생

따라서 **DB 상태 전이를 먼저 커밋하고, 외부 호출은 이후에 수행**합니다.

==== 스케줄러 호출: `NotificationJobScheduleEvent`로 분리

외부 스케줄러 호출은 `NotificationJobScheduleEvent`를 통해 분리됩니다.
이벤트는 상태 전이 TX 안에서 발행되고, Spring Modulith의 `@ApplicationModuleListener`가 TX 커밋 후 별도 TX에서 소비합니다.
`NotificationJobScheduleProcessor`가 이벤트를 받아 `NotificationJobSchedulePort`를 호출합니다.

[cols="2,2,2,3", options="header"]
|===
| 발행 주체 | 이벤트 action | 소비 동작 | 소비 실패 시

| `NotificationJobCreatedProcessor`
| `REGISTER`
| `schedulePort.schedule()`
| Job은 SCHEDULED이지만 스케줄 row 없음. +
발송 트리거 안 됨 → 수동 복구 필요

| `CancelNotificationJobUseCase`
| `CANCEL`
| `schedulePort.cancelSchedule()`
| 고아 스케줄 잔존. +
Relay가 도달해도 Job이 CANCELLED이므로 `markProcessing()` 실패 → 스킵

| `NotificationJobSendOrchestrator`
| `RETRY`
| `schedulePort.retrySchedule()`
| 자동 재시도 미등록. +
Job은 FAILED이므로 `RecoverNotificationJobUseCase`로 수동 복구 가능
|===

==== 발송 호출: `NotificationService.send()` — 트랜잭션 안에서 실행

`NotificationService.send()`(외부 채널 발송)는 트랜잭션 안에서 호출됩니다.
이는 의도적인 설계입니다:

* `NotificationSendingStatePersister`가 SENDING 상태를 **별도 `REQUIRES_NEW` TX**로 먼저 커밋
* 외부 발송은 `@ApplicationModuleListener`의 TX 안에서 실행
* 발송 결과(SENT/FAILED)와 Job 최종 상태(COMPLETED/FAILED)가 원자적으로 커밋

발송을 TX 밖으로 빼면 "발송 성공 + DB 반영 실패" 시 복구가 더 어려워집니다.
현재 구조에서는 발송과 결과 저장이 같은 TX에 있으므로, 크래시가 아닌 한 일관성이 유지됩니다.

**트레이드오프:** 외부 발송 응답 대기 동안 **Job row의 행 잠금**이 유지됩니다.
알림 수가 많거나 외부 채널 응답이 느리면 잠금 보유 시간이 길어질 수 있습니다.
이는 다른 실행 주체(StuckRecovery, CancelUC 등)가 해당 Job에 접근할 때 대기가 발생한다는 의미입니다.

**TX 중첩 구조:**

[source]
----
@ApplicationModuleListener TX (REQUIRES_NEW) — Job row FOR UPDATE 잠금 보유
├── sendingStatePersister.persist() — REQUIRES_NEW TX (독립 커밋)
│   └── Notification rows: PENDING/RETRY_WAITING → SENDING 저장
├── sender.send() — 외부 채널 발송
├── applySendResults() — SENDING → SENT/FAILED 저장
├── applyClassificationResults() — FAILED → RETRY_WAITING/DEAD_LETTER 저장
└── resolveJobStatus() — Job: PROCESSING → COMPLETED/FAILED
    └── NotificationJobScheduleEvent(RETRY) 발행 (TX 안에서 발행, 커밋 후 소비)
----

==== UC의 락-트랜잭션 순서

UseCase(`Cancel`, `Recover`)는 `TransactionTemplate`을 사용하여
**트랜잭션 커밋 → 분산 락 해제** 순서를 보장합니다.

[source]
----
try {
    transactionTemplate.execute(status -> {
        // DB 작업 (상태 전이, Notification 변경)
    });
    // ← TX 커밋 완료
} finally {
    distributedLock.unlock(...);
    // ← 락 해제 — 다른 실행 주체가 진입해도 커밋된 상태를 읽음
}
----

`@Transactional` 대신 `TransactionTemplate`을 사용하는 이유는,
`@Transactional`에서는 `finally`의 락 해제가 메서드 반환 전에 실행되어
TX 커밋보다 먼저 락이 풀리는 창(window)이 생기기 때문입니다.

**예외: `CreateScheduledNotificationJobUseCase`** -- TX 커밋 **전에** 락을 해제합니다.
이는 `NotificationJobCreatedAfterCommitEvent`가 TX 커밋 후 `@ApplicationModuleListener`로 소비될 때,
`NotificationJobCreatedProcessor`가 같은 `idempotencyKey`로 분산 락을 잡아야 하기 때문입니다.
TX 커밋 전 락 해제로 인한 레이스 윈도우는 `NotificationJobCreatedProcessor`가 CREATED 상태인지 확인하는 것으로 방어됩니다.

=== 이벤트 체인

Processor가 다른 이벤트를 발행하는 경우 이벤트 체인이 형성됩니다:

* `NotificationJobCreatedAfterCommitEvent` → *NotificationJobCreatedProcessor* → `NotificationJobScheduleEvent(REGISTER)`

=== 최초 실행 트리거 흐름

`DbScheduledNotificationJobRelay`는 SCHEDULED → PROCESSING 전이만 수행하고 `NotificationJobExecutionEvent`는 발행하지 않습니다.
최초 실행은 다음 흐름으로 트리거됩니다:

1. `DbScheduledNotificationJobRelay`: SCHEDULED → PROCESSING 전이 + `NotificationJobStatusChangedEvent` 발행
2. `ProcessingNotificationJobExecutionScheduler`: PROCESSING Job을 주기적으로 폴링 (5초 간격)
3. PROCESSING 상태 감지 → `NotificationJobExecutionEvent` 발행
4. `NotificationJobExecutionProcessor`: 이벤트 수신 → `SendOrchestrator.execute()` 위임
